# Исследование: десктопное приложение вставки текстовых шаблонов (Snipcast)

**Целевые платформы:** macOS и Windows. Поддержка Linux в объёме продукта не планируется.

## 1. Цель продукта

Приложение для быстрой вставки заранее подготовленных фрагментов текста в активное окно (или в буфер обмена) по глобальной горячей клавише. Интерфейс вызова — в духе **Raycast**: компактное плавающее окно, поиск сверху, список результатов с заголовком и превью, опциональная иерархия (подпункты), внизу подсказки по клавишам. Приложение постоянно в **системном трее**; основное окно не держится на экране постоянно.

## 2. Референс: что взять из Raycast

| Аспект | Описание для нашего приложения |
|--------|--------------------------------|
| **Триггер** | Глобальный shortcut → мгновенное появление окна поверх других приложений, фокус в поле поиска. |
| **Поиск** | Фильтрация списка по заголовку и, при необходимости, по телу шаблона; приоритет совпадений в названии. |
| **Строки списка** | Заголовок (крупнее/жирнее) + одна строка «содержания» (обрезка с многоточием). |
| **Подпункты** | Группа: один логический шаблон → несколько вариантов (например, «Подпись» → «Краткая» / «Полная») или вложенные категории; навигация стрелками, Enter — вставка выбранного. |
| **Подсказки внизу** | Статическая или контекстная строка: Enter — вставить, Esc — закрыть, ⌘↵ — копировать без вставки и т.д. (под платформу подписи меняются: Ctrl/⌘). |
| **Визуал** | Тёмная/светлая тема, скругления, отступы, клавиатурная навигация без мыши — ожидание пользователя от «raycast-подобного» UX. |

## 3. Функциональные требования (сводка)

- **Трей**: иконка, меню (настройки, выход), без обязательного главного окна в доке/панели задач (по желанию — скрытие из дока на macOS).
- **Горячая клавиша**: настраиваемая комбинация, работающая когда фокус в любом приложении.
- **Два типа шаблонов**:
  - **Мастер (read-only)**: корневая папка задаётся в настройках; содержимое только читается приложением; редактирование через файловую систему вне приложения (или только админом).
  - **Свои**: CRUD в UI, отдельное хранилище (файлы или БД).
- **Переменные** в тексте: плейсхолдеры вида `{user}`, `{mail}` и др.; при вставке подстановка из профиля пользователя (и, опционально, из окружения/конфига).

## 4. Поведение вставки

Варианты реализации на разных ОС различаются по надёжности:

1. **Эмуляция ввода** (симуляция набора текста) — работает во многих полях, но медленнее и чувствительна к раскладке/IME.
2. **Буфер обмена** — скопировать итоговый текст + симулировать Cmd+V / Ctrl+V — часто самый практичный компромисс на десктопе.
3. **Только копирование** — без автоматической вставки (проще с правами и безопасностью).

Имеет смысл заложить **настраиваемое поведение**: «вставить в активное окно» vs «только в буфер».

## 5. Модель данных шаблонов

### 5.1 Файловое представление (мастер-папка)

- Один файл = один шаблон, или один JSON/YAML на группу с подпунктами.
- Рекомендация для простоты синхронизации и Git: **Markdown или plain text** с frontmatter (YAML) для заголовка, тегов, порядка сортировки.
- Подпункты: либо подпапки с `index` + файлы, либо один файл со списком секций.

Пример frontmatter:

```yaml
---
title: Приветствие
preview: Добрый день, {user}...
order: 10
---
Полный текст шаблона...
```

### 5.2 Пользовательские шаблоны

- Та же схема метаданных, хранение в user data directory (см. раздел про стек).
- Редактор: встроенный простой редактор или открытие внешнего редактора по кнопке.

### 5.3 Переменные

- Словарь подстановок: встроенные ключи (`user`, `mail`, `date`, `time`) + пользовательские поля в настройках.
- Синтаксис: `{key}`; опционально фильтры позже (`{date|YYYY-MM-DD}`) — не обязателен в MVP.
- Не найденный ключ: оставить плейсхолдер или пустую строку — зафиксировать в настройках.

## 6. Стек для macOS и Windows: варианты

### 6.1 Tauri 2 + Rust + веб-UI (React/Vue/Svelte)

**Плюсы:** небольшой размер, нативный трей и глобальные shortcuts через плагины/крейты, доступ к файловой системе.

**Минусы:** глобальный хоткей и симуляция вставки требуют аккуратной интеграции по платформам (macOS / Windows).

**Оценка:** хороший баланс для «лёгкого» десктоп-приложения с Raycast-подобным UI на HTML/CSS.

### 6.2 Electron

**Плюсы:** зрелая экосистема, много примеров tray + globalShortcut.

**Минусы:** больший размер и потребление RAM; глобальные хоткеи и вставка в другое приложение всё равно упираются в OS APIs и иногда в права доступности (macOS).

### 6.3 Flutter desktop

**Плюсы:** единый UI.

**Минусы:** системный трей и глобальные хоткеи — через плагины, качество по платформам неоднородно; вставка в стороннее приложение — отдельная задача.

### 6.4 Две нативные оболочки + общая логика (Rust core + Swift/Kotlin)

**Плюсы:** максимально «родное» поведение на macOS и Windows.

**Минусы:** дороже в поддержке; для соло или малой команды обычно избыточно.

### 6.5 Выбор для Snipcast

Реализация ведётся на **Tauri 2** + React/TypeScript: нативный трей, плагин глобальных горячих клавиш, компактный бинарник. Для визуальной близости Raycast используется веб-стек (кастомная вёрстка списка и подсказок).

## 7. Платформенные нюансы

### macOS

- **Accessibility**: для программной вставки (симуляция Cmd+V) часто нужны разрешения «Универсальный доступ» / Accessibility.
- **Окно поверх всех**, **не показывать в доке** — штатные настройки NSWindow / Tauri window API.
- Горячие клавиши: не пересекаться с системными; показать конфликт в UI.

### Windows

- Глобальный хук клавиатуры или RegisterHotKey; антивирусы иногда чувствительны к низкоуровневым хукам — предпочтительны API уровня приложения, где возможно.
- Автовставка через буфер + SendInput/simulate paste.

## 8. Безопасность и приватность

- Шаблоны могут содержать персональные данные — хранение только локально по умолчанию.
- Мастер-папка: только чтение снижает риск случайной порчи корпоративных шаблонов.
- Логи не должны печатать полный текст шаблонов без явной отладочной опции.

## 9. Конкуренты и ориентиры (для дальнейшего UX-бенчмарка)

- **Raycast** — UX поиска и списка (не текстовые сниппеты как основной кейс, но паттерн UI).
- **Alfred**, **LaunchBar** — сниппеты и буфер.
- **Espanso** — кроссплатформенная подстановка текста по триггерам (другая модель: триггер-слово vs палитра).
- **aText**, **TextExpander** — коммерческие аналоги.

Отличие целевого продукта: **палитра по hotkey + мастер/свои + переменные + raycast-стиль**.

## 10. MVP: предлагаемый объём

1. Трей + одна настраиваемая глобальная горячая клавиша.
2. Окно поиска: фильтр, список с заголовком и превью, клавиатурная навигация, подсказки внизу.
3. Мастер-папка (read-only) + пользовательская папка или встроенное хранилище.
4. Подстановка `{user}`, `{mail}` и 2–3 встроенных полей из настроек профиля.
5. Действие по Enter: «вставить через буфер» или «только копировать».
6. Подпункты: минимальный вариант — вложенные элементы в одном JSON/файле или flatten-список с отступом в UI.

## 11. Риски

| Риск | Митигация |
|------|-----------|
| Вставка не работает в некоторых приложениях | Режим «только копирование» + уведомление. |
| Конфликт hotkey | Детекция и предупреждение в настройках. |
| Большие списки шаблонов | Виртуализация списка, debounce поиска. |

## 12. Следующие шаги после исследования

1. Зафиксировать целевые ОС и минимальные версии (например, macOS 13+, Windows 10+).
2. Стек: **Tauri 2** — прототип: окно + hotkey + чтение папки.
3. Прототип подстановки переменных и вставки на macOS и Windows.
4. UI-kit / дизайн-токены под «raycast-подобный» вид (тёмная тема по умолчанию).
